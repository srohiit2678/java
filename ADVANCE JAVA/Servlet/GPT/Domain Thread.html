Runtime Exception vs. Error in Java

Both RuntimeException and Error are subclasses of Throwable, but they serve different purposes.

1. RuntimeException (Unchecked Exception)

  • Definition: A subclass of Exception that represents problems due to programming logic errors.
  • When It Occurs: During program execution (at runtime).
  • Handling: Can be caught and handled using try-catch.
  • Examples:
    • NullPointerException Accessing a null reference
    • ArrayIndexOutOfBoundsException Accessing an array index that doesnt exist
    • ArithmeticException Division by zero
    • ClassCastException Invalid type casting

Example:

java
public class RuntimeExceptionExample {
    public static void main(String[] args) {
        String str = null;
        System.out.println(str.length()); // Throws NullPointerException
    }
}

2. Error

  • Definition: Represents severe issues that the application should not try to catch.
  • When It Occurs: Due to system-level failures (e.g., JVM issues, resource exhaustion).
  • Handling: Typically not caught because they indicate serious problems.
  • Examples:
    • StackOverflowError Infinite recursion
    • OutOfMemoryError JVM runs out of memory
    • VirtualMachineError JVM crash

Example:

java
public class ErrorExample {
    public static void recursiveMethod() {
        recursiveMethod(); // Infinite recursion -> StackOverflowError
    }

    public static void main(String[] args) {
        recursiveMethod();
    }
}

What is a Daemon Thread?

A Daemon Thread is a low-priority thread in Java that runs in the background to perform tasks like garbage collection, monitoring, or other system-level tasks.

Key Feature: Daemon threads automatically terminate when all non-daemon threads (user threads) finish execution.

User Threads vs. Daemon Threads

FeatureUser ThreadDaemon Thread
DefinitionA normal thread that performs core application tasksA background thread that supports user threads
LifespanRuns until it completesTerminates when all user threads finish
PriorityUsually higherLower priority
UsageHandling user requests, business logic, calculationsGarbage collection, background monitoring
ExampleThreads in a web server processing requestsJVMs garbage collector

Creating a Daemon Thread

A thread can be converted into a daemon thread using the setDaemon(true) method before it starts.

Example:

java
class DaemonThreadExample extends Thread {
    public void run() {
        while (true) {
            System.out.println("Daemon thread is running...");
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }
    
    public static void main(String[] args) {
        DaemonThreadExample daemonThread = new DaemonThreadExample();
        daemonThread.setDaemon(true); // Set as daemon thread
        daemonThread.start();

        System.out.println("Main thread is running...");
        try {
            Thread.sleep(3000); // Main thread sleeps for 3 seconds
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        
        System.out.println("Main thread finished execution!");
        // After this, JVM terminates the daemon thread automatically
    }
}

Key Points

  1. Daemon threads must be set before starting (setDaemon(true) must be called before start()).
  2. They automatically stop when all user threads finish.
  3. They are useful for background tasks like logging, monitoring, and garbage collection.
  4. Not recommended for critical tasks because they may terminate unexpectedly.

Default Daemon Threads in JVM

When a Java application starts, several daemon threads are automatically started by the JVM to handle background tasks. Some of the important ones include:

  1. Garbage Collector (GC) Thread

    • Responsible for automatic memory management.
    • Runs in the background and removes unused objects from memory to free up space.
    • Example: Finalizer thread in older Java versions.

  2. Finalizer Thread

    • Calls the finalize() method on objects before they are garbage collected.
    • Helps clean up resources like file handles, database connections, etc.
    • In newer Java versions (Java 9+), finalize() is deprecated in favor of try-with-resources.

  3. Reference Handler Thread

    • Manages weak, soft, and phantom references.
    • Used for memory-sensitive caching and garbage collection optimizations.

  4. Signal Dispatcher Thread

    • Handles OS-level signals sent to the JVM.
    • Example: CTRL+C in the terminal sends a shutdown signal to Java programs.

  5. Attach Listener Thread

    • Used for dynamic debugging and profiling.
    • Allows tools like JVisualVM and JConsole to attach to a running JVM instance.

Checking Background (Daemon) Threads in Your JVM

You can check all running threads, including daemon threads, using this code:

java
public class ListJVMThreads {
    public static void main(String[] args) {
        ThreadGroup rootGroup = Thread.currentThread().getThreadGroup();
        while (rootGroup.getParent() != null) {
            rootGroup = rootGroup.getParent();
        }

        Thread[] threads = new Thread[rootGroup.activeCount()];
        rootGroup.enumerate(threads, true);

        System.out.println("JVM Threads:");
        for (Thread t : threads) {
            System.out.println("Thread Name: " + t.getName() + " | Daemon: " + t.isDaemon());
        }
    }
}

Example Output:

yaml
JVM Threads:
Thread Name: main | Daemon: false
Thread Name: Reference Handler | Daemon: true
Thread Name: Finalizer | Daemon: true
Thread Name: Signal Dispatcher | Daemon: true
Thread Name: Attach Listener | Daemon: true

Key Takeaways

  1. Daemon threads run in the background and support application functionality.
  2. JVM automatically starts several daemon threads for garbage collection, memory management, debugging, and OS interactions.
  3. User threads keep the application alive, while daemon threads terminate automatically when no user threads are running.
  4. You can list and analyze these threads using ThreadGroup.

Great question! Let's dive into Weak, Soft, and Phantom References in Java. These references are part of Javas memory management system and help the Garbage Collector (GC) efficiently manage memory.

1. Strong Reference (Default)

Before getting into weak, soft, and phantom references, let's understand strong references, which are the default in Java.

java
String str = new String("Hello");
  • Here, str is a strong reference to the String object.
  • As long as str exists, the object wont be garbage collected.

2. Weak Reference (WeakReference)

  • Allows the object to be garbage collected even if it is still referenced.
  • Used for caches where objects should be removed when memory is low.

Example:

java
import java.lang.ref.WeakReference;

public class WeakReferenceExample {
    public static void main(String[] args) {
        String strongRef = new String("Weak Reference");
        WeakReference<String> weakRef = new WeakReference<>(strongRef);

        System.out.println("Before GC: " + weakRef.get()); // Weak Reference

        strongRef = null; // Remove strong reference
        System.gc(); // Suggest GC to run

        System.out.println("After GC: " + weakRef.get()); // Likely null (collected)
    }
}

Use Case: Caches (e.g., WeakHashMap) where objects should not prevent garbage collection.

3. Soft Reference (SoftReference)

  • Similar to weak references but more reliable.
  • GC only collects soft-referenced objects when memory is really low.
  • Used for caching mechanisms where objects should stay as long as theres memory.

Example:

java
import java.lang.ref.SoftReference;

public class SoftReferenceExample {
    public static void main(String[] args) {
        String strongRef = new String("Soft Reference");
        SoftReference<String> softRef = new SoftReference<>(strongRef);

        System.out.println("Before GC: " + softRef.get()); // Soft Reference

        strongRef = null;
        System.gc();

        System.out.println("After GC: " + softRef.get()); // May be null if memory is low
    }
}

Use Case: Image or object caching in applications (e.g., web browsers, memory-sensitive apps).

4. Phantom Reference (PhantomReference)

  • Unlike weak/soft references, phantom references do not return the object using .get().
  • Used to perform cleanup actions before GC removes an object.
  • Requires a ReferenceQueue.

Example:

java
import java.lang.ref.PhantomReference;
import java.lang.ref.ReferenceQueue;

public class PhantomReferenceExample {
    public static void main(String[] args) {
        String strongRef = new String("Phantom Reference");
        ReferenceQueue<String> queue = new ReferenceQueue<>();
        PhantomReference<String> phantomRef = new PhantomReference<>(strongRef, queue);

        System.out.println("Before GC: " + phantomRef.get()); // Always null

        strongRef = null;
        System.gc();

        System.out.println("After GC: " + (queue.poll() != null ? "Object is GC'd" : "Not yet GC'd"));
    }
}

Use Case: Tracking object finalization and releasing external resources (like closing files, database connections).

Key Takeaways

  1. Weak references: Objects can be garbage collected when not strongly referenced (used in caches).
  2. Soft references: Objects persist longer but are collected when memory is low (used in memory-sensitive caching).
  3. Phantom references: Used for cleanup actions before GC collects the object (tracking object finalization).
  4. Default references in Java are strong references, meaning objects stay in memory until explicitly dereferenced.
